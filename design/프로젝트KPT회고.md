# 토이프로젝트Ⅰ  2조 KPT 회고

## 회고 개요
- [KPT 회고](https://brunch.co.kr/@jinha0802/35) 를 바탕으로 이번 토이프로젝트1  2조의 KPT회고를 정리함.
- Keep: 현재 만족하고있고, 계속 이어갔으면 하는 부분
- Problem: 불편하게 느끼는 부분, 개선이 필요한 부분
- Try: Problem에 대한 해결책
- Practice: Try 중에서 당장 실행 가능한것. 또는 구체적인 Try예시

## KEEP

### 코드 리뷰 및 문제점 공유
- 모든 팀원이 PR할 때 함께 코드 리뷰를 진행했고, 문제점이 있을 때 이를 공유하여 빠르게 해결했습니다.
### 커뮤니케이션
- 애자일 스프린트를 하면서 매일 만나 회의를 진행하고 친목을 쌓았습니다.
### 프로젝트 기획
- 컨벤션, 스타일 가이드 및 진행 방식을 설정하고 시작해서 코드 통일성을 보장했습니다.

### 페어 프로그래밍
- 문제점 해결이나 다른 코드를 병합할 때 팀원들 간 페어 프로그래밍을 진행했습니다.
### 공통 스타일 가이드 구축
- 디자인 스타일 가이드를 정해놔서 일관적인 페이지 구성을 할 수 있었습니다.
- [토이프로젝트1 팀2 스타일 가이드 · Issue #189 · Dev-FE-1/team2-intranet-project- · GitHub](https://github.com/Dev-FE-1/team2-intranet-project-/issues/189)
### 자료 공유
- 팀원 간 유용한 정보를 찾으면 꾸준히 공유했습니다.

### 이슈 활용
- 서로 일정을 확인할 수 있게 기록하고 커밋을 하여 프로젝트 관리를 진행했습니다.    
- 이슈 템플릿을 세팅한 것.
    
### 스토리보드 피드백을 통한 프로젝트 이해도 상승
- 왜 이 기능을 사용했는지 이유를 명확히 했었다.
- 클래스 컴포넌트로 개발한 이유가 명확함. 클래스 컴포넌트로 개발했을 때의 장점 등.
### 디렉토리 구조를 정하고 간 것
- 미리 디렉토리를 구조를 정해두어 어떤 파일을 어디에 넣어야될지 쉽게 알 수 있었습니다.
### 질문을 하기 좋은 팀 분위기
- 팀원 간 자유로운 질문과 답변이 오가는 분위기를 조성했습니다.
    

## PROBLEM & TRY

###  일정 관리 미흡
#### problem
- QA를 제대로 하지 못했고 발표 준비도 제대로 하지 못했습니다.
#### try
- 협업툴(Jira, 깃허브 프로젝트)를 활용해서 일정 관리하기
- buffer 기간 잡기 : 기능 구현 시간의 1.5~2배
- End-to end 테스트 도입
#### practice
 - 프로젝트 기간이 5주라면 개발기간은 3주로 잡기
- 예시) [Playwright](https://playwright.dev/) EtoE 테스트 도구 이용해서 버튼이 잘눌리는지 클릭할 때 오류가 발생하는지 전체 웹사이트가 잘 작동 되는지 테스트하기 



### 코드 병합과 코드 통합의 지연

#### problem
- 좀 더 일찍 페이지를 병합했으면 좋았을 것 같습니다. 병합 과정에서 에러가 많이 발생하여 마지막 주에 테스트를 하지 못해 아쉬웠습니다
#### try
- 프로젝트 병합을 미리 계획하여 에러를 줄이고, 충분히 테스트할 수 있도록 계획하기

#### practice
- 개발 기간이 5주였다면 2주차에 병합을 진행을 한다. 
- 병합은 이번주 수요일에 붙이도록하자.
    
### 프로젝트에 대한 목적과 문제 정의 미흡

#### problem
- 인트라넷 서비스의 사용자가 어떤 기능을 원할지 등에 대한 명확한 정의 없이 개발을 진행했습니다.
- 프로젝트 필수 요구사항을 제대로 숙지 안하고 프로젝트를 진행함.

#### try
- 유저 페르소나 정하기: 해당 유저의 성향과 유저가 누구인지를 유저 특정해서 해당 유저가 필요한 요구사항들을 만드는 것이 좋을 것 같다.
#### practice
- 저희 프로젝트는 핀테크 개발 회사의 회사원들과 관리지가 사용하는 인트라넷을 구축 할 것입니다.
    

### 유저 동선 미흡

#### problem
- 웹사이트의 유저 동선을 고려하지 않았습니다. 예를 들어, 관리자 직원 관리 페이지에서 직원 클릭 시 상세 페이지로 이동한 후 다시 돌아가기 위한 뒤로가기 버튼이 없었습니다.
#### try
- 기능 구현 전에 유저 동선을 생각해보기
#### practice
- 관리자 유저 상세 확인 기능 동선 확인하기 <br/>
 `관리자 직원 로그인 -> 관리자 유저 직원 관리 페이지 -> 유저 상세 확인 페이지 ->취소 버튼 누름 -> 다시 관리자 유저 직원 페이지로 이동.`

### 부자연스러운 UI

#### problem
- 올바른 아이디를 입력하면 안내 문구가 빨간색이 아니라 녹색으로 나타나도록 하여 유저가 헷갈리지 않게 해야 했습니다.
#### try
- Ul를 사용자에 맞게 개선하여 안내 문구 색상 등을 개선하기
#### pratice
- 경고문구는 빨간색, 성공 문구는 파란색으로 수정하기

### 효율적인 회의 활용 부족

#### problem
- 회의를 효율적으로 활용하지 못한 점이 있었습니다.
- 회의록을 빠짐 없이 적기 힘들었음.
- 영양가 없는 회의가 3시간 넘게 오래 진행되는 일도 있었다.
#### try
- 회의 공유 목록 가이드 정하기
- 제한시간을 정하고, 그 안에서 인원별 시간 분배하기
- 회의 기록 남기기 툴 활용하기  
#### pratice
- 프로젝트 인원이 5명이라면 1시간을 제한을 두고, 한 사람당 12분 정도 쓰게 시간을 분배하기
-  [Main || 마음회의록](https://minutes.maum.ai/) (또는 [클로바노트](https://clovanote.naver.com))가 회의 기록하고 회의록을 작성하게 하기 



### 요구사항과 기능 정의서 관리 부족

#### problem
- 변경된 사항들에 대해 기능정의서와 요구사항 정의서 등 문서에 기록하지 않은 점이 있었습니다.
#### try
- 프로젝트 진행 중 변경사항이 생기면 문서에 기록하여 관리하기
- 동기화해서 확인할 수 있는 공유문서 툴 적극 활용

#### pratice
- 갤러리 페이지 네이션 기능을 무한 스크롤로 수정했다면 노션의 기능 정의서와 피그마 스토리보드도 간단하게 라도 수정하기

### 기록으로 남기는 코드 리뷰 없이 PR 진행

#### problem
- PR을 진행하면서 코드 리뷰를 기록으로 남기지 않은 점이 있었습니다.
#### try
- 깃허브 플로우 전략을 효율적으로 활용하기 위해 PR을 할 때마다 팀원 간에 코드 코멘트를 작성하여 리뷰를 하기
- 페어 리뷰어를 정하고 해당 리뷰어와 페어로 코드 리뷰를 진행한다.
#### pratice
- A님 B님과 함께 이번 페이지네이션 컴포넌트 구현 코드에 대해 페어로 코드 리뷰 진행해주세요.
